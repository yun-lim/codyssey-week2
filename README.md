# 나만의 퀴즈 게임 (Console Quiz Game)

## 프로젝트 개요
터미널(콘솔) 환경에서 동작하는 퀴즈 게임 프로그램입니다. Python의 기본 문법과 객체 지향 프로그래밍(클래스)을 활용하여 구현하였으며, 게임 데이터는 JSON 파일로 영구 저장됩니다. Git을 활용한 버전 관리와 함께, 프로그램의 구조화, 파일 입출력, 예외 처리를 실습합니다.

## 퀴즈 주제와 선정 이유
- 주제: 영어 구동사(Phrasal Verbs) 표현 학습
- 선정 이유: 영어 구동사를 학습하고 있어, 퀴즈 형태로 반복 학습하면서 자연스럽게 표현을 익히기 위해 선정했습니다.

## 실행 방법
사전 요구 사항: **Python 3.10 이상** (외부 라이브러리는 필요하지 않습니다.)

터미널을 열고 프로젝트 폴더로 이동한 뒤 아래 명령어를 입력하여 프로그램을 실행합니다.

```bash
python3 main.py
```

## 주요 기능 목록

### 필수 기능
1. **퀴즈 풀기**: 저장된 퀴즈를 랜덤 순서로 출제합니다. 풀 문제 수를 선택할 수 있으며, 정답/오답 여부를 판별하고 최종 점수를 출력합니다.
2. **퀴즈 추가**: 새로운 퀴즈 문항(문제, 4개 선택지, 정답 번호, 힌트)을 직접 등록할 수 있습니다.
3. **퀴즈 목록**: 현재 등록된 모든 퀴즈 목록을 조회합니다.
4. **점수 확인**: 최고 점수와 전체 게임 기록 히스토리를 확인합니다.
5. **종료**: 데이터를 저장하고 프로그램을 종료합니다.

### 보너스 기능
- **랜덤 출제**: 퀴즈 풀기 시 문제 순서가 무작위로 섞입니다.
- **문제 수 선택**: 풀고 싶은 문제 수를 직접 선택할 수 있습니다.
- **힌트 기능**: 풀이 중 0번을 입력하면 힌트를 볼 수 있습니다. (힌트 사용 시 0.5점 감점)
- **퀴즈 삭제**: 등록된 퀴즈를 선택하여 삭제할 수 있습니다.
- **점수 히스토리**: 모든 게임 기록(날짜/시간, 문제 수, 점수)이 저장됩니다.

### 예외 처리
- 숫자가 아닌 입력, 범위 밖 입력, 빈 입력에 대한 안내 메시지 출력 후 재입력
- Ctrl+C / EOF 발생 시 데이터 저장 후 안전 종료
- state.json 파일이 없거나 손상된 경우 기본 데이터로 복구

## 설계 및 클래스 구조

### 클래스 책임 분리
본 프로젝트는 역할에 따라 두 개의 클래스로 분리하여 설계했습니다.

- **Quiz 클래스**: 개별 퀴즈 한 문제를 표현하는 데이터 단위입니다.
  - 속성: `question`(문제), `choices`(선택지 4개), `answer`(정답 번호), `hint`(힌트)
  - 메서드: `display()`(문제 출력), `check_answer()`(정답 확인), `to_dict()`/`from_dict()`(JSON 변환)
  - 책임: 퀴즈 데이터의 보관과 표현만 담당하며, 게임 흐름이나 저장에는 관여하지 않습니다.

- **QuizGame 클래스**: 게임 전체의 흐름과 상태를 관리하는 컨트롤러입니다.
  - 속성: `quizzes`(퀴즈 목록), `best_score`(최고 점수), `score_history`(기록)
  - 메서드: `show_menu()`, `play_quiz()`, `add_quiz()`, `list_quizzes()`, `show_score()`, `delete_quiz()`, `save_data()`, `load_data()`, `run()`
  - 책임: 사용자 입력 처리, 게임 진행, 데이터 저장/불러오기 등 프로그램 전체 흐름을 담당합니다.

### 로직 분리 기준
기능을 메서드로 분리할 때 다음 기준을 적용했습니다.

1. **하나의 메서드 = 하나의 역할**: 메뉴 표시(`show_menu`), 퀴즈 출제(`play_quiz`), 데이터 저장(`save_data`) 등 각 메서드가 한 가지 기능만 수행합니다.
2. **입력 검증 공통화**: 숫자 입력 검증(`_get_number_input`)과 텍스트 입력 검증(`_get_text_input`)을 별도 메서드로 추출하여 중복 코드를 제거했습니다.
3. **데이터와 로직 분리**: Quiz 클래스는 데이터 표현만, QuizGame 클래스는 게임 로직만 담당하도록 분리했습니다.

## 기술 원리

### 클래스를 사용하는 이유 (vs 함수만 사용)
함수만으로도 프로그램을 만들 수 있지만, 클래스를 사용한 이유는 다음과 같습니다.

- **데이터와 동작의 묶음**: 퀴즈의 문제, 선택지, 정답이라는 데이터와 출력, 정답 확인이라는 동작이 논리적으로 하나의 단위입니다. 함수만 사용하면 이 데이터를 매번 딕셔너리로 전달해야 하지만, 클래스를 사용하면 `quiz.check_answer(3)`처럼 자연스럽게 호출할 수 있습니다.
- **상태 관리**: QuizGame은 퀴즈 목록, 최고 점수 등 프로그램 상태를 유지해야 합니다. 클래스의 인스턴스 변수(`self.quizzes`, `self.best_score`)를 사용하면 전역 변수 없이 상태를 관리할 수 있습니다.
- **확장성**: 새로운 기능(삭제, 힌트 등)을 추가할 때 해당 클래스에 메서드만 추가하면 되므로 구조를 변경하지 않고 확장할 수 있습니다.

### JSON을 데이터 저장에 사용하는 이유
- **사람이 읽을 수 있는 형식**: state.json 파일을 텍스트 에디터로 열어 직접 확인하고 수정할 수 있습니다.
- **Python 표준 라이브러리 지원**: `json` 모듈이 기본 내장되어 있어 외부 라이브러리 없이 바로 사용할 수 있습니다.
- **딕셔너리/리스트와 자연스러운 변환**: Python의 `dict`, `list` 자료형과 JSON 구조가 거의 동일하여 변환이 간단합니다.
- **경량성**: 소규모 데이터(퀴즈 수십 개, 점수 기록)를 저장하기에 별도의 데이터베이스 없이 파일 하나로 충분합니다.

### 파일 입출력에서 try/except가 필요한 이유
파일 작업은 프로그램이 제어할 수 없는 외부 요인에 의해 실패할 수 있습니다.

- **파일이 존재하지 않는 경우** (`FileNotFoundError`): 첫 실행 시 state.json이 없으므로 기본 데이터로 시작해야 합니다.
- **파일이 손상된 경우** (`json.JSONDecodeError`): 사용자가 직접 파일을 편집하거나, 저장 중 프로그램이 종료되면 JSON 형식이 깨질 수 있습니다.
- **디스크 오류** (`OSError`): 디스크 공간 부족, 권한 문제 등으로 쓰기가 실패할 수 있습니다.

try/except 없이는 이런 상황에서 프로그램이 에러 메시지와 함께 비정상 종료되지만, try/except를 사용하면 오류를 감지하고 기본 데이터로 복구하거나 사용자에게 안내 메시지를 보여줄 수 있습니다.

## 데이터 파일 설명 (`state.json`)

### 데이터 구조와 설계 이유
모든 퀴즈 데이터와 점수 정보는 프로젝트 루트 디렉터리에 위치한 `state.json` 파일에 저장됩니다 (UTF-8 인코딩). 프로그램 재시작 시 해당 파일을 읽어와 이전 상태를 복구합니다.

```json
{
    "quizzes": [
        {
            "question": "\"look up\"의 의미로 가장 적절한 것은?",
            "choices": ["찾아보다", "올려다보다", "포기하다", "돌아보다"],
            "answer": 1,
            "hint": "모르는 단어가 있을 때 사전에서 ___합니다."
        }
    ],
    "best_score": 80,
    "score_history": [
        {
            "date": "2026-04-10 14:30:00",
            "total": 5,
            "correct": 4,
            "hints_used": 1,
            "score": 80,
            "is_best": true
        }
    ]
}
```

**각 필드의 설계 이유:**
- **`quizzes`** (리스트): 퀴즈가 순서 있는 목록이므로 리스트로 관리합니다. 각 퀴즈는 `question`, `choices`, `answer`, `hint`를 가진 딕셔너리입니다.
- **`answer`** (정수 1~4): 정답을 문자열이 아닌 번호로 저장하여 비교 로직을 단순화했습니다.
- **`best_score`** (정수 또는 null): 아직 게임을 하지 않은 상태(`null`)와 0점을 구분하기 위해 null을 허용합니다.
- **`score_history`** (리스트): 최고 점수만으로는 학습 추이를 파악할 수 없으므로 모든 게임 기록을 저장합니다. `is_best` 플래그로 최고 점수 갱신 시점을 표시합니다.

### 데이터 읽기/쓰기 흐름
```
프로그램 시작
  └→ load_data() 호출
       ├→ state.json 파일 열기 (open, json.load)
       ├→ 성공: 퀴즈 목록/점수를 메모리에 로드
       ├→ FileNotFoundError: 기본 퀴즈 데이터로 시작
       └→ JSONDecodeError/기타: 안내 메시지 출력 후 기본 데이터로 초기화

게임 진행 중 (메모리에서 데이터 조작)
  ├→ 퀴즈 추가/삭제 시: save_data() 호출
  └→ 퀴즈 풀기 완료 시: 점수 기록 추가 후 save_data() 호출

프로그램 종료
  ├→ 정상 종료 (메뉴 6번): save_data() 호출 후 종료
  └→ 비정상 종료 (Ctrl+C/EOF): try/except로 감지 → save_data() 호출 → 안전 종료
```

`save_data()`는 `json.dump()`로 전체 데이터를 한 번에 파일에 덮어씁니다. 데이터 변경이 발생할 때마다 즉시 저장하여 프로그램이 예기치 않게 종료되어도 데이터 손실을 최소화합니다.

## 예외 처리 상세

### 입력 예외 처리
- **빈 입력** (Enter만 입력): 안내 메시지 출력 후 재입력 요청
- **숫자가 아닌 입력** (예: "abc"): `int()` 변환 시 `ValueError` 발생 → 안내 메시지 출력 후 재입력
- **범위 밖 숫자** (예: 메뉴에서 9): 범위 검사 후 안내 메시지 출력 → 재입력

### Ctrl+C / EOF 안전 종료 처리
프로그램의 메인 루프(`run` 메서드)를 `try/except (KeyboardInterrupt, EOFError)`로 감싸서 처리합니다.

```python
def run(self):
    try:
        while True:
            self.show_menu()
            # ... 게임 루프
    except (KeyboardInterrupt, EOFError):
        self.save_data()
        print('\n\n    프로그램을 안전하게 종료합니다.')
```

사용자가 Ctrl+C를 누르거나 입력 스트림이 종료되면, 현재 진행 중인 데이터를 `save_data()`로 파일에 저장한 뒤 안내 메시지를 출력하고 정상적으로 프로그램을 종료합니다. 이를 통해 추가한 퀴즈나 점수 기록이 유실되지 않습니다.

### 파일 손상 시 복구 전략
state.json이 손상된 경우(JSON 파싱 실패, 필수 키 누락, 타입 불일치) 다음과 같이 대응합니다.

1. `json.load()` 시 `JSONDecodeError` 발생 → 기본 퀴즈 데이터 7개로 초기화
2. 필수 키(`quizzes`) 누락 시 `KeyError` 발생 → 기본 데이터로 초기화
3. 데이터 타입 불일치 시 `TypeError` 발생 → 기본 데이터로 초기화
4. 사용자에게 "저장 파일이 손상되어 기본 퀴즈 데이터로 초기화합니다." 안내 메시지 출력
5. 이후 정상적으로 게임 진행 가능하며, 다음 저장 시 정상 파일로 덮어씌워짐

**한계**: 현재는 손상 시 기존 데이터를 모두 버리고 초기화합니다. 백업 파일을 만들거나 부분 복구를 시도하는 방식으로 개선할 수 있습니다.

## 브랜치 전략
퀴즈 풀기 기능은 `feature/play-quiz` 브랜치에서 개발한 후 main에 병합했습니다.

- **브랜치 분리 이유**: 새로운 기능을 개발하는 동안 main 브랜치의 안정성을 유지하기 위해서입니다. 기능이 완성되기 전에 main에 직접 커밋하면, 미완성 코드가 포함된 상태가 됩니다.
- **병합(merge)의 의미**: 분리된 브랜치에서 완성한 기능을 main에 합치는 과정입니다. 이를 통해 main은 항상 동작하는 상태를 유지할 수 있습니다.
- **실무 활용**: 팀 프로젝트에서는 각자 feature 브랜치에서 작업한 뒤, 코드 리뷰(PR)를 거쳐 main에 병합하는 방식으로 협업합니다.

## JSON 저장 방식의 한계와 개선 방향
현재 JSON 파일 저장 방식은 소규모 데이터에는 적합하지만 다음과 같은 한계가 있습니다.

- **데이터 증가 시 성능 저하**: `json.dump()`는 매번 전체 데이터를 파일에 덮어씁니다. 퀴즈가 수천 개 이상으로 늘어나면 저장/불러오기 시간이 길어집니다.
- **동시 접근 불가**: 두 명 이상이 동시에 프로그램을 실행하면 파일 덮어쓰기로 인해 데이터가 유실될 수 있습니다.
- **부분 수정 불가**: 퀴즈 하나만 추가해도 전체 파일을 다시 써야 합니다.
- **검색/필터링 비효율**: 특정 조건의 퀴즈를 찾으려면 전체 데이터를 메모리에 로드한 뒤 순회해야 합니다.

**개선 방향**: 데이터가 대규모로 늘어난다면 SQLite 같은 경량 데이터베이스로 전환하여 부분 읽기/쓰기, 인덱스 검색, 동시 접근 등을 지원할 수 있습니다.

## 요구사항 변경 시 수정 포인트
향후 요구사항이 변경될 경우 수정해야 할 부분을 정리합니다.

| 변경 시나리오 | 수정 파일 | 수정 내용 |
|--------------|-----------|-----------|
| 선택지 개수 변경 (4개→5개) | `quiz.py` | `_get_answer_with_hint()`의 범위(1~4→1~5), `add_quiz()`의 입력 루프, Quiz 클래스 문서 |
| 새 메뉴 항목 추가 | `quiz.py` | `show_menu()`에 항목 추가, `run()`에 분기 추가, `_get_number_input` 범위 변경 |
| 저장 형식 변경 (JSON→SQLite) | `quiz.py` | `save_data()`/`load_data()` 메서드를 DB 연동으로 교체, `Quiz.to_dict()`/`from_dict()`는 유지 가능 |
| 멀티플레이어 지원 | `quiz.py` | 사용자별 점수 관리 로직 추가, state.json 스키마에 사용자 식별 필드 추가 |
| 카테고리별 퀴즈 분류 | `quiz.py` | Quiz 클래스에 `category` 속성 추가, state.json 스키마 변경, 메뉴에 카테고리 필터 추가 |

클래스 기반 설계 덕분에 대부분의 변경은 해당 클래스의 메서드 수정이나 추가로 대응할 수 있으며, 프로그램 전체를 재작성할 필요가 없습니다.
